# iter 10 — date typing, panel-flash fix, no-suggestion note, prompt v3

**Date:** 2026-05-06
**Branch:** `feat/document-prefill`

---

## Context

Four issues surfaced after iter 9:

1. **Date pickers are click-only.** Custom-period start and end dates ([Assessment.tsx:1618-1700+](src/pages/Assessment.tsx#L1618)) use a shadcn `<Popover>` + react-day-picker `<Calendar>` only. To pick FY 1995 the user has to scroll the calendar back ~360 months. There is no input field. The user wants to be able to type `30/06/1995` directly.

2. **Explanation panel flashes briefly on no-suggestion answers.** Iter 9 set the outer panel gate to `requires_explanation || docsCount > 0`. That fires on every answer when the session uploaded any docs, even on questions where (a) the answer doesn't require explanation and (b) no AI prefill exists. Net effect: a grey "💡 Explanation" panel + textarea briefly appears as the user clicks "No" on such a question, then auto-advance whisks them away. Visual noise + violates the iter-6 fast-click-through promise.

3. **No subtle indicator on no-suggestion questions.** When a session has docs but a particular question has no prefill, the user wants a small playful line to acknowledge it: "*No suggestion for this one. You're on your own here.*"

4. **Prompt v2 still produces meta-language about documents.** Example: *"The VDD specifically notes for the German entities that S4 DE BV and S4 DE GP BV are limited taxpayers in Germany following the conversion of the German GmbHs into KGs..."*. v2's TONE rule banned "the documents", "the memorandum", "the local file" but didn't enumerate other doc-types ("VDD", "VDR", "the report", "the financials", "the jaarrekening"). The model leaks via these. Bump to v3 with a stricter blanket rule + concrete bad/good example.

This iteration addresses all four.

---

## Approach

### A. Date typing alongside the calendar

**File:** [src/pages/Assessment.tsx](src/pages/Assessment.tsx) — both period date fields (start and end), around lines 1618-1700.

Hybrid pattern: a typeable text `<Input>` for `dd/mm/yyyy` next to the existing `<Popover>` calendar button (now a small `<CalendarIcon>` button only). Either path works:

- **Type path:** user types `30/06/1995` → onBlur parses with `parse(value, "dd/MM/yyyy", new Date())`. If `isValid` → store as `yyyy-MM-dd`. If invalid → set field-level error, leave value as-is for editing.
- **Click path:** click the calendar icon → popover opens → pick → fills the input via the same setter.

Layout becomes:
```
[Label]    [ dd/mm/yyyy            ] [📅]
           ^ Input — typeable        ^ small button popping up the Calendar
```

Both date fields (`period_start_date`, `period_end_date`) get the same treatment. Existing validation in `handleStartSession()` (start ≤ end) is unchanged. The button label text "30/06/1995 ▼" disappears since the input itself shows the value.

### B. Panel gate revert

**File:** [src/pages/Assessment.tsx](src/pages/Assessment.tsx) outer gate around line 2068.

Change from:
```tsx
selectedQuestionOption?.requires_explanation || docsCount > 0
```
to:
```tsx
selectedQuestionOption?.requires_explanation || !!currentPrefill
```

That is the original iter-8 rule. Panel renders on (a) explanation-required answers (existing) OR (b) any AI prefill present for this question (regardless of `suggested_toelichting` content, so a prefill with only `answer_rationale` still surfaces the panel). On a `requires_explanation = false` answer with no prefill → no panel ever, no flash.

The empty-state line *"No relevant context found in the uploaded documents for this question."* inside the panel becomes unreachable now and is removed. Item C below replaces that role with a subtler line OUTSIDE the panel.

### C. Subtle no-suggestion line

**File:** [src/pages/Assessment.tsx](src/pages/Assessment.tsx) — render below the answer-row block, alongside the existing iter-7 plain-rationale strip.

Render only when:
- `docsCount > 0` (session uploaded ≥1 doc),
- `prefillJob?.status === "completed"` (analysis done; we don't taunt during in-flight),
- `!currentPrefill` (no AI output for this question),
- `selectedAnswer` (user picked something — line shouldn't appear pre-selection).

Copy:
> *No suggestion for this one. You're on your own here.*

Style: `text-xs italic text-muted-foreground mt-3 ml-1 mb-3` — same family as the rationale strip but with playful tone.

### D. Prompt v3 — stricter meta-language ban

**File:** new migration `supabase/migrations/20260506100000_swarm_prompt_v3.sql`.

Replace the v2 TONE rule (rule 1) with a stricter version:

> 1. ADVISOR FIRST-PERSON VOICE WITH HEDGED INFERENCE. Speak as the advisor typing their own toelichting. NEVER reference any document by name or category — banned phrases include but are not limited to: "the documents", "the memorandum", "the memo", "the local file", "the master file", "the report", "the VDD", "the VDR", "the financials", "the jaarrekening", "the analysis", "according to...", "based on...", "the analysis covers...", "as noted in...", "the [doc type] notes/states/says/specifies/indicates that". The general rule: NEVER say or imply you are reading from a document. Speak as if YOU have direct knowledge of these facts.
>    - When evidence is fact-dense and explicit (named parties, jurisdictions stated, numbers cited): state directly. Example: "Camden B.V. is a Dutch BV that...".
>    - When the conclusion involves an inference — drawn from absence of mention, indirect derivation, or partial evidence: hedge AT the conclusion, inside the advisor voice. Use "It seems that...", "Likely...", "Appears to be...", "Based on the indications,...", "There do not appear to be...". Do NOT hedge by pointing at documents — hedge the conclusion itself.
>
>    BAD example: "The VDD specifically notes for the German entities that S4 DE BV is a limited taxpayer in Germany."
>    GOOD example: "S4 DE BV is a limited taxpayer in Germany following the conversion of the German GmbHs into KGs. It seems this concerns the partners' limited tax liability via partnership transparency rather than a Dutch head office operating a foreign branch."

Bump version to 3, deactivate v2, set `notes` describing the change. Apply via `az vm run-command` (same pattern as the v2 migration applied 2026-05-04).

## Out of scope

- Replacing the calendar with a fully native `<input type="date">` (browser-locale variance, not consistent across users).
- Allowing the user to type the year via dropdowns inside the calendar (alternative path; rejected because typing the full date is faster than dropdown + click).
- Exporting the prompt-content management to the admin UI (already exists via PrefillPrompts admin page; new version still seeds via SQL migration).
