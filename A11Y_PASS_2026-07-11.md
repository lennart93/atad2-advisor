## Open decisions for Lennart

1. **Contrast tokens (the one remaining axe/Lighthouse failure, on every
   page).** Four token-level failures against WCAG 1.4.3 (4.5:1 for normal
   text). Minimal darkened values that keep hue/saturation:

   | Token | Now | Ratio (worst bg) | Minimal AA fix |
   |---|---|---|---|
   | `--muted-foreground` (hsl 40 8% 51% = #8c8578) | 3.24 on #f2f1ed, 3.45 on page | fails everywhere it renders ≤ 17 px | **#736d62** (safe on all three bgs) |
   | `--ds-ink-tertiary` #8a8479 | 3.50 on page | same family | **#777268** |
   | `--ds-accent` #c25c3c as small text (stepper counters, 11px terracotta labels) | 4.05 on page | only small-text uses fail | **#b65638**, or bump those labels to large/bold |
   | `--ds-accent-text` #a8492d on `--ds-accent-bg` #f0ddd5 (answer badges) | 4.38 | just under | **#a4472c** (imperceptible) |

   Reference: `--ds-ink-secondary` #57534a passes at 7.2:1, so an alternative
   is mapping the failing muted text onto the existing secondary token.
   I have not changed any colour; this is the palette from the A–D campaign.

2. **Assessment wizard is `min-w-[1024px]` by design** (AssessmentShell), so
   the wizard itself horizontally scrolls at 320 px/200 % zoom and the
   structure step shows "best viewed on a wider screen" below ~lg. True
   WCAG 1.4.10 reflow for the wizard means a responsive redesign — out of
   scope under the "don't alter layout" guardrail. Decide: accept as a
   documented exception (the chart itself qualifies for the 2-D exemption,
   the question/worklist steps strictly do not) or schedule a responsive pass.

3. **The `html { zoom: 1.15 }` rule taxes every breakpoint** (a media query
   fires at 640 px but only ~556 px of layout space exists). I compensated
   the one concrete failure (dashboard ledger); any future 200 %-zoom issues
   will keep tracing back to this rule. Consider retiring the zoom in favour
   of real sizes at some point.

4. **Test credentials.** The audit signed in with the password you provided
   in chat; consider rotating it. For the browser-verification of protected
   routes to be repeatable (CI or future passes), a dedicated low-privilege
   test account would help — creating one on the production Supabase was
   deliberately not done autonomously.
